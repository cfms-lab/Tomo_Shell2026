> **[해결 — 2026-07-04 새벽, Tomo_Shell2026-dev 쪽 수정·검증 완료]**
>
> **근본 원인 (이슈 A·B 공통)**: 07-01/02에 들어온 슬롯 삽입 경로
> (`cu_insertSlotPixel_SetBit`)가 ① 슬롯별 **글로벌 스핀락** + 락 안 64엔트리 선형
> 중복검사(이슈 B의 ~130× 직렬화), ② **용량 64 포화 시 어떤 픽셀이 살아남는지가
> GPU 스케줄링 순서에 좌우**(이슈 A의 비결정성·0-셀)였음. E3 1°에서 비어있지 않은
> 슬롯의 87%가 포화 상태였고, 픽셀의 58%가 조용히 버려지고 있었다
> (`TOMO_DEBUG_SLOT_INPUT` sat/rows 카운터로 확인).
> 멀티스트림 가설은 기각 — `TOMO_CUDA_STREAMS=1`에서도 동일 재현.
>
> **수정 내용** (`cfms_tomo/Tomo_Shell_vc_src/cuda_src/`):
> 1. 삽입을 **락-프리 CAS-클레임 오픈어드레싱**으로 재작성 — (z,nZ)를 키 하나로
>    패킹해 `atomicCAS`로 엔트리 클레임, 중복은 `atomicOr` 타입 병합. 수집 용량
>    64→512 (`CU_SLOT_CAPACITY_16`).
> 2. step2 직전 **결정적 절단 패스**(`cu_truncateSlots`) 추가 — 엔트리별 최소 삽입
>    삼각형 ID(atomicMin)를 기록해 "삼각형 순서상 먼저 온 63개"만 유지. CPU
>    (`STomoVoxelSpaceInfo`, 슬롯 63픽셀)와 같은 체제를 결정적으로 재현.
>    z-우선 절단은 vss −25% 편향이 생겨 기각함. `CU_SLOT_TRUNCATE_TO`로 조절
>    (=CU_SLOT_CAPACITY_16이면 전량 유지 모드 — 단 CPU보다 vss ~2× 커짐).
> 3. `reducedSum_Streamed`의 **cudaEvent 누수 수정** (방향당 2개 생성 후 미파괴,
>    1° 스윕당 ~26만 개; 생성 실패 시 동기화가 조용히 사라지는 잠재 레이스).
> 4. 미사용이지만 깨져 있던 `atomicOr(char*)`(atomicWrite.cu) CAS 루프로 수정.
> 5. step2 블록 (256×8)→(512×2), 정렬 패스를 슬롯 실제 길이 기준으로 단축.
>
> **검증 (RTX 5080, E3=Group_E3_931902 1,050,398면)**:
> - 취약 방향 18개 × 10회 반복: **완전 일치** (전 방향 distinct=1), 스퓨리어스 0-셀 0개.

## 이슈 A — 산발적 0 반환 (조용한 커널 실패) + 런 간 비결정성

### 증상 (E3 = g5test/Group_E3_931902.stl, 1,050,398면, 1°/60° 스윕 2회)

- 두 런 모두 **정확히 7개 셀이 vss=0.0** (이웃 셀은 28~336 정상값). 0이 그리드
  최솟값이 되어 "최적 방향"으로 뽑히는 것이 실질 피해.
- 0-셀 좌표 (yaw idx, pitch idx):
  - run1: (0,323) (1,323) (2,323) (177,141) (179,145) (180,143) (360,323)
  - run2: (0,323) (180,143) (183,141) (355,320) (358,323) (359,324) (360,323)
  - 공통 3개, 나머지는 **같은 두 근방 안에서 셀만 이동** → 결정적 취약 영역 + 확률적 발현.
- **기하학적 규칙성**: 두 근방은 (yaw≈0/360°, pitch≈323°=360−37°)과
  (yaw≈180°, pitch≈143°=180−37°). yaw 0↔360은 동일 방향(랩어라운드)이고,
  (yaw+180°, 180°−pitch)는 같은 물리 방향의 대칭 표현 — 즉 **사실상 단일 물리
  방향(≈−37° 계열)의 표현들에서만 터진다.** 특정 회전각에서 이 메시의 어떤 기하가
  경합 또는 수치 특이점을 만드는 것.
- 비결정성은 0-셀만이 아님: 0 아닌 130,310셀 중 두 런 완전 일치가 **20,773개뿐**
  (최대 차이 831, 전 그리드 max ~36,000 대비 ~2.3%). CPU(INT3)는 결정적.

### 코드 단서 (cfms_tomo/Tomo_Shell_vc_src/cuda_src/)

- `CUDA_types.cuh:34`에 `_CUDA_USE_ATOMIC_WRITE_FOR_INT32_SLOTBUFFER` 정의됨 →
  int32 슬롯 쓰기는 atomic. 그러나:
- **`atomicWrite.cu:31 atomicOr(char*)`가 의심스러움**: 내부에서
  `long_val = __byte_perm(long_old, 0, long_address_modulo) + val;` (셀렉터 자리에
  modulo를 넘기고 OR 대신 `+`) 후 `atomicOr(base_address, replacement)`를 호출하고
  `long_old != long_assumed` 루프로 재시도 — CAS가 아니라 OR라 재시도 의미가 없고
  이웃 바이트 오염 가능. char 단위 타입 플래그 버퍼에 쓰인다면 레이스+오염 경로.
- 멀티스트림: `STomoSh_CUDA.cu` nStream = min(SM수, 8), 스트림별 슬롯 버퍼를
  나눠 쓰는 구조 — 버퍼 재사용/동기화 누락 시 방향 간 오염 가능.
  **`TOMO_CUDA_STREAMS` 환경변수(STomoSh_CUDA.cu:412)로 스트림 수 강제 가능** —
  1로 고정해 0-셀·비결정성이 사라지는지 보는 것이 1차 실험.
- 대안 가설(비-레이스): yaw 0/360°·특정 pitch에서 회전행렬/복셀화(step1_voxelize.cu)의
  수치 경계(예: 정확히 축 정렬된 삼각형, fmod 경계) → 해당 방향에서 복셀이 전혀
  생성되지 않아 합계 0. "정확히 0"이라는 점은 이쪽과도 부합. 두 가설 판별:
  스트림 1개·동일 방향 단건 반복 호출 시 0이 재현되면 수치 경계, 사라지면 레이스.

### 수정 판정 기준

- E3 1°/60° 스윕 2회 반복에서 0-셀 0개 **그리고** 두 런 그리드 완전 일치(또는
  부동소수 tolerance 이내). SFTF 쪽 저장 그리드(run1/run2)와 비교 스크립트 있음.

## 이슈 B — 대형 메시 ~130× 슬로다운

### 증상 (모두 같은 밤, 유휴, 동일 dll)

| 메시 | faces | CUDA 1°/60° | 비고 |
|---|---|---|---|
| Group_E9_82675 | 916,548 | 2,084 s | |
| Group_E2_790253 | 1,039,452 | 4,284 s | |
| Group_E4_439142 | 1,038,714 | 12,392 s | |
| Group_E3_931902 | 1,050,398 | 20,037 s / 19,943 s (2회) | 재현 안정 |
| Group_E8_64445 | 514,938 | **27,646 s** | **과거 기록 212 s → ~130×** |

- E8의 3°/60° 계열 5개(θc=0/30/45/60/90°)는 각각 3,081.9~3,084.5s로 **극히 균일**
  → 방향당 비용 ≈0.211s로 고정된 상태 (130,321방향 × 0.211 ≈ 27,600s와 정확히 일치).
  즉 스윕 오버헤드가 아니라 **방향당 처리 비용 자체가 커진 것**.
- 면수와 비용이 비례하지 않음 (E9 92만면 2,084s vs E8 51만면 27,646s, 13×) —
  **면수가 아니라 특정 기하(복셀 밀도? 슬롯 픽셀 수?)에 지배됨**.
- 소형 메시는 정상 (Bunny 69k 3° CUDA 등 기존 기록과 유사).
- 참고: CPU(INT3)는 같은 메시들에서 2,000~2,900s(1°) — CUDA가 CPU보다 10배 느린 상황.

### 조사 제안

1. 과거 212s 기록을 낸 구버전 dll과의 diff (파이프라인 변화, 배치 크기, 스트림 수,
   슬롯 버퍼 크기·재할당 여부).
2. 방향당 스테이지 타이머 (step1 voxelize / step2 pairing / step3 bed / step4 reduce)
   — E8 vs E9 비교로 병목 스테이지 특정. `tomoDebugPrintCudaSlotInput`(STomoSh_CUDA.cu:41)
   의 sat/maxLen 카운터가 이미 있음: **슬롯 포화(sat>0)나 maxLen 폭주면 슬롯 용량
   초과 재시도/직렬화 경로가 유력**.
3. `TOMO_CUDA_STREAMS=1` vs 8 비교 (스트림 경합이면 차이 큼).
4. Nsight Systems 프로파일 (H2D/D2H 복사가 방향마다 반복되는지, 커널 점유율).

### 수정 판정 기준

- E8 1°/60°가 수백 초 스케일로 복귀 (과거 212s 수준), E2~E9도 CPU 대비 우위 회복.
